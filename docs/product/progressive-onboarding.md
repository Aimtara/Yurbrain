# Progressive onboarding

Last updated: 2026-04-30

Yurbrain onboarding should explain the continuity loop progressively. The product should feel like a backup brain that helps a person capture, resurface, continue, convert, act, and return — not like a generic task manager or chat app.

## Stage 1 — Capture + Focus Feed

First-run copy should emphasize:

- capture quickly,
- find the thought again,
- continue later without needing to decide now.

Primary UI language:

- “Capture now. Find it again when it matters.”
- “Save it now, find it again, and continue later.”
- “Capture quickly. Yurbrain will bring it back when it can help you continue.”

The primary capture action is **Save**. Planning is available, but secondary.

## Stage 2 — Plan This contextually

Plan This should appear when the user is already looking at a resurfaced item, item detail, or comment that naturally suggests conversion to action.

It should not be framed as the mandatory next step for every capture. Copy should say “turn this resurfaced thought into a next step” rather than “create a task.”

## Stage 3 — Return, postpone, rebalance

Revisit Later, postpone, and rebalance language belongs after the product has evidence of incomplete, blocked, paused, or deferred work.

Recommended copy:

- “Return later without losing context.”
- “Choose when this should come back.”
- “Break into a smaller step” for repeatedly postponed work.

## Advanced concepts

Do not introduce Liquid Time, Open Loops, or founder-specific execution vocabulary in first-run empty states. Existing lens labels may remain where they already support navigation, but first-run/help text should use plain continuity language.

## Launch stance

For the current web-first launch scope, progressive onboarding is intentionally light-touch inline copy, not a wizard. A heavier onboarding flow should be a separate product decision after staging smoke and founder feedback.
